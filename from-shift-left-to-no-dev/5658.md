# Context

This page continues the [pbc-4445][3] / [pbc-4849][5] / [pbc-5051][6] / [pbc-5253][7] / [pbc-5455][8] series. The same Darmok scenario ran on two cycles against `sheep-dog-grammar`: `Rebuild56` (2026-05-26 late evening) and `Rebuild58` (2026-05-27 evening). Both runs are **post the harness fix** that [pbc-5455][8] diagnosed (commit `abc5e3f`, `Precompute JaCoCo coverage shortlist in GreenPhase, reference from identify.md`) — the shortlist producer now writes its artifact on every scenario, observable as 12 occurrences of `Green: Writing jacoco-shortlist.md` in the 2026-05-26 mojo log and 17 in 2026-05-27. That regression case is closed.

The pair-range topped the gsheet tab (`gid=886919682`) for the run window with a different scenario — `This object step definition text parameter exists with other parameters validation`, green delta 229,852 ms (~3:50). Inspection isolates two findings:

1. **A new identify-phase variance** (152 s, ~66% of the gap) driven by a [Claude Code Glob](#) tool quirk: when given `pattern=<absolute path>`, the tool finds files inside gitignored directories; when given `pattern=<relative>` + `path=<projectRoot>`, the gitignore filter trips and the same file returns "No files found." `identify.md` has no explicit no-error path, so each run improvises a different verification — one form found the file, the other didn't, and the slower run spent 196 s reasoning about a missing-but-present jacoco.xml and even launched a `mvn verify` to "repopulate" it.
2. **A 78 s residual fix-phase variance** of the same shape as [pbc-4849][5] / [pbc-5455][8]: search-strategy drift before the same single edit lands in `TextIssueDetector.java`. Even with the shortlist now populated, claude's pre-edit exploration still spans different breadths.

| | Case |
|---|---|
| Scenario | `This object step definition text parameter exists with other parameters validation` |
| Rebuild56 commit | `6cadef46` |
| Rebuild58 commit | `17e5b05f` |
| Green-phase range | **229,852 ms (~3:50)** — top of `gid=886919682` |

Unlike [pbc-5455][8], where both final commits were identical, run-56 and run-58 patched **different lines of the same file** with structurally different patches — both valid for the failing assertion. The fix-phase signature here is [pbc-4445][3]-shaped (different patches, both valid) overlaid on the [pbc-4849][5] / [pbc-5455][8] pre-edit-exploration cost. The identify-phase signature is new.

# Root Cause

Two contributors, in priority order.

## (1) `identify.md` has no explicit no-COMPILATION-ERROR exit

`identify.md` lines 17–23:

```
Before doing any other work, call TodoWrite once with this exact list,
then mark each item completed as you finish it:

1. Grep ${logPath} for COMPILATION ERROR|FAILED|ERROR|Tests run.*Failures|BUILD FAILURE; do NOT read the full file.
2. If COMPILATION ERROR is present:
   a. Read both ${umlDir}/uml-interaction-main.md and ${umlDir}/uml-interaction-test.md.
   b. Create stub classes under ${projectPath}/src/test/java/org/farhan/impl/...
   c. Loop mvn verify -Dtest=${runnerClassName} until only non-compile failures remain (jacoco.xml populated).
```

When there is no COMPILATION ERROR, the prompt provides no step 3 — no "exit, you're done" instruction. Both runs improvise an extra check ("let me at least confirm jacoco.xml is there"). The improvisation is free to use any tool form claude prefers. R56 picked an absolute-pattern Glob; R58 picked a relative-pattern + `path=` Glob; the two forms return different things on a gitignored target — see (2).

The prompt also mentions in its opening paragraph: *"The baseline check before this run and the Phase Verification check after each fix guarantee that ${runnerClassName} is the only failing test."* That invariant *should* let identify exit cleanly on the no-error path — but it's a parenthetical, not a step. Claude reads the steps and improvises around them.

## (2) Claude Code Glob tool: absolute-pattern bypass vs `path=`-arg gitignore filter

Verified directly against the file currently on disk (same path the runs were querying):

```
$ stat /home/farhan/.../target/site/jacoco-with-tests/jacoco.xml
size: 522280, mtime: 2026-05-27 18:02:12 -0400
$ grep -E "^target|/target" /home/farhan/git/sheep-dog-main/.gitignore
**/target
```

The file exists. Repo-root `.gitignore` has `**/target`.

| Glob form | Behavior | Why |
|---|---|---|
| `pattern="/abs/path/to/file/in/target/foo.xml"` (no `path`) | Finds the file (R56 form) | Absolute pattern is user-asserted scope; gitignore not applied. |
| `pattern="target/foo.xml" path="/abs/projectRoot"` | Returns `"No files found"` (R58 form) | `path` + relative pattern engages the project-scoped glob mode that respects `.gitignore`; the `**/target` rule excludes the match. |

This is a discoverable, reproducible quirk in how Claude Code's `Glob` tool treats the `path` argument. The tool documentation does not call out the gitignore behavior on the `path=` form. It is a real foot-gun whenever the prompt is asking claude about a build-output file under a gitignored directory.

R58 spent 196 s in identify; R56 spent 44 s. The 152 s delta is **purely** the cost of this Glob-form quirk meeting the silent no-error path in `identify.md`.

## Why two contributors and not one

Either fix on its own would close the case-study gap:
- If `identify.md` had an explicit *exit immediately on no COMPILATION ERROR*, both runs would skip call 5 onward. No Glob, no quirk, no detour.
- If the Glob tool didn't filter on `path=`, both runs would have got `numFiles: 1` and exited at call 6 the same way.

The harness can't fix the Glob tool (it's a Claude Code internal behavior). The harness *can* fix the prompt, so that's where the remediation sits.

## Why this wasn't caught by [pbc-5455][8]'s fix

[pbc-5455][8] fixed the shortlist producer so `fix.md`'s precondition is met. Both runs in this case study confirm the producer is working: each shows `Green: Writing jacoco-shortlist.md` in the mojo log and a successful `Read jacoco-shortlist.md` returning content in the JSONL. That fix is in place.

What [pbc-5455][8] did not touch is `identify.md`'s no-error path. The identify-phase fix-up was out of scope of the producer bug, but it's the next layer down — the same kind of "prompt under-specification" the [pbc-4849][5] / [pbc-5455][8] series has been peeling off.

Place: **prompt** (with a tool-quirk amplifier). The fix lives in `identify.md`.

# The Fix

Two prompt patches, tracked as **[#414][9]** (identify.md) and **[#415][10]** (fix.md). Both apply the same pattern: the prompt currently *describes* what the system provides without issuing the corresponding *imperative*, leaving claude to improvise — and improvisations differ across runs.

## (1) `identify.md` — explicit no-COMPILATION-ERROR exit ([#414][9])

```
3. Otherwise (no COMPILATION ERROR), identify is complete — exit immediately.
   Do not Glob, Read, or Bash to confirm jacoco.xml or other preconditions;
   the mojo's baseline check before this prompt and the precompute hook after
   it are the contracts.
```

This closes the silent no-error branch and names the disallowed actions explicitly. "Do not Glob, Read, or Bash to confirm preconditions" is the load-bearing clause — without it, a thorough claude might still improvise a precondition check ("just to be sure") before exiting. Naming the disallowed actions makes *exit* mean *exit now*.

The identify-phase fix doesn't help every scenario in the case-study window — `This object step definition text parameter doesn't exist validation` had a clean identify phase on both runs and 100% fix-phase variance. The identify fix tackles the cases where claude improvised; the fix-phase variance is the next layer down. (Confirmed on three sibling scenarios scanned during case-study follow-up: identify improvisation appears sporadically, not universally.)

## (2) `fix.md` — explicit "read every listed class before any Grep/Glob" ([#415][10])

The follow-up scan that surfaced the fix-phase shape (sibling scenario `text parameter doesn't exist validation`, +122s, all fix-side) traced to a parallel under-specification in `fix.md` step 1:

> Read `${shortlistPath}` first: if populated it narrows your search to the classes other passing tests in this family already exercise plus the production classes those tests cover. The sibling test classes show you the shape the failing test's code path should take; the production classes show you the code path the passing tests exercise.

The literal imperative is *read the shortlist file*. The prose describes what the listed classes contain. There's no instruction to *read those classes* — claude is left to infer "and now go read them" and picks its own search method. R56 read 7 sibling files end-to-end; R58 grep/glob'd the interface hierarchy symbol-by-symbol. The pattern the shortlist exists to surface lives in *file structure*, not in indexable symbols — whole-file reads converge faster than symbol grep on this codebase.

Proposed step 1 rewrite:

```
1. Read ${shortlistPath} first. Then read every production class listed in
   the shortlist end-to-end before any other Read, Grep, or Glob. The sibling
   test classes show you the shape the failing test's code path should take;
   the production classes show you the code path the passing tests exercise.
   Do not Grep or Glob for symbols, interfaces, or method names until after
   the listed classes are read — the pattern lives in file structure, not
   in indexable symbols.
```

Same anti-improvisation shape as the identify.md patch. Related: [#411][11] proposes expanding the TodoWrite list to enumerate the shortlist files; the two changes compose (TodoWrite enumerates the files; step-1 wording enforces the order).

## (3) Defense-in-depth (out of scope for this case)

The mojo could write the prompt with absolute paths interpolated for `${jacocoPath}` already, removing the temptation to invent the relative-pattern + `path=` Glob form. Candidate for a separate sweep across all darmok prompt templates.

The Glob tool's `path=`-arg gitignore filtering is silently inconsistent with the absolute-pattern form. Not Anthropic's bug to fix from this repo, but worth a `pbc-tool-quirks.md` entry so future case studies don't re-discover it. The tool returns "No files found" with no indication of *why*; the runner-log + JSONL pair is the only way to spot it externally.

Place for both prompt fixes: **prompt — producer (mojo distributes the prompt resource at `target/darmok/identify.md` and `target/darmok/fix.md`)**. Single file each. No spec change, no test change, no tool-side change required.

# What This Validates

1. **The methodology layers cleanly.** [pbc-5455][8] fixed the deepest cause (producer never wrote its artifact). This case finds the next layer up (the prompt that wraps the artifact has a silent branch). Once that branch is closed, the next pair-range fire will name the layer above *that*, and so on. Pair-range is a "what's the loudest assignable cause left" filter that survives upstream fixes.
2. **Glob argument form is a meaningful diagnostic axis.** The first three cases in the series differed on *which command claude typed* (different grep regexes, different file Glob patterns). This case differs on **which form of the same tool call claude used** — same target, same intent, same result-shape-by-name, different result-by-internal-filter. Future case studies should record the full tool input, not just the tool name and target file, when comparing divergent calls.
3. **The mojo's precompute can be made trustworthy from the prompt side.** [pbc-5455][8]'s fix made the precompute *correct*; this case's fix makes the prompt *believe* the precompute. Both are necessary. A producer that works but is verified-against by the consumer still produces variance; a prompt that trusts a working producer eliminates the verify-against step entirely.
4. **Identical files, different patches is a real configuration of the pair-range signal.** Prior cases had either identical commits or different files. This case is the same file, different *spots in the file* — and pair-range still fired with the expected magnitude. The methodology generalizes past the "different files" shape.
5. **Refactor variance is a sibling, not a child, of green variance.** R56 spent 194 s longer on refactor than R58 for this scenario; the case study isolates green (per the gsheet's column choice) but the refactor delta is its own pair-range signal, on a different tab. The series is well-positioned to add a refactor-phase tab to the sheet next, using the same JSONL methodology.

# Open Questions From This Case

- After [#414][9] and [#415][10] land, the next Rebuild pair on this scenario should show identify wall-clock collapse to ~30–50 s on both sides (one TodoWrite + one Grep + UML reads + done — UML reads are now unconditional per [#411][11]'s direction) and fix-phase pair-range compress as the symbol-grep improvisation gives way to ordered whole-file reads. If either doesn't move, the prompt changes weren't the right knob.
- The refactor delta on this scenario (R56 was 194 s slower than R58) is itself worth a follow-up case study on a refactor-phase-targeted pair-range tab. R56's refactor made three extra Edits (import normalization, identifier shortening, control-flow restructure) that R58 did not — possibly because R58's first-Edit shape didn't need those follow-ups. Whether that's a spec interpretation gap or a model-choice difference is unknown without inspecting the `/rgr-refactor` JSONL portions.
- A `pbc-tool-quirks.md` entry on `Glob`'s `path=` + gitignore interaction would prevent re-discovery. The same quirk presumably affects any darmok prompt that asks claude to verify `target/` artifacts; identify.md is just the first one to surface it.
