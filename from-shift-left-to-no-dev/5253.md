# Context

This page continues the [pbc-4445][3] / [pbc-4849][5] series. The same Darmok scenario ran on consecutive cycles against `sheep-dog-grammar`, both on 2026-05-25, on branches `Rebuild52` and `Rebuild53`. The pair-range was flagged outside the upper control limit on the gsheet (`flag: out`) — the only such row on this tab — so this is the cleanest assignable-cause signal in the recent series. The diagnosis traces to a different prompt-template site than [pbc-4849][5]: not a missing JaCoCo annotation, but a missing **sibling-scenario** guardrail in `fix.md`.

| | Case |
|---|---|
| Scenario | `This object step definition text parameter doesn't exist validation` |
| Rebuild52 commit | `9cb99b9b` |
| Rebuild53 commit | `95ba4ac9` |
| Green-phase range | **3.0 min** (178,631 ms) — outside UCL (170,071 ms) |
| Files patched (final, post-refactor) | R52: 5 files, R53: 4 files (R52 added an enum value; R53 reused an existing one) |
| Diagnosed cause | prompt — `fix.md` does not direct Claude to check the negation-paired sibling Test-Cases that share the runner's `@RGR` tag |

The sibling list is already intrinsic to the runner: the runner class is tag-driven (`@RGR`), every Test-Case in the runner carries that tag, and the matching `.feature` file (which both runs read) lists all nine siblings in one block. The remediation is therefore a behavioral instruction in `fix.md`, not a new precompute file.

## The build-retry cycle

R52's first batch of 5 edits compiled. The first `mvn clean verify` came back with:

The failing test is the **sibling** scenario `text parameter exists validation`, not the TC the green phase is implementing. R52's new code path returned a `Present` (issue-detected) result for any `IText` cursor, regressing the case where the step parameter Content **does** exist.

R52 then ran two Greps (`expected:|but was:|AssertionFailedError` in `log.txt`; `text parameter exists|text parameter doesn't exist` in the .feature), Read the .feature for the sibling-row Step-Parameters table, and made one corrective Edit to `TestStepIssueDetector.java` adding the Content-parameter guard. R53's first batch of 4 edits already included the guard: it located the existing `validateStepParameterListWorkspace(IRow)` method (which has the canonical "if Content parameter exists, return empty" pattern) and **factored a shared helper** `getStepDefinitionForTestStep(ITestStep)` out, then wrote the new `validateTextWorkspace(IText)` against the same helper, applying the same Content guard.

# Root Cause

The sibling-shortlist already exists implicitly. The runner class is **tag-driven** — `RGRTest` runs every Test-Case carrying `@RGR`, and the matching `.feature` file lists all nine siblings in a single block. Verified by `grep -nE "^  Scenario:" "src-gen/test/resources/cucumber/specs/Language Definition/Issues/Validation for Workspace Issues.feature"`:

```
 10:  Scenario: This object doesn't exist validation
 26:  Scenario: This object step definition doesn't exist validation
 44:  Scenario: This object step definition parameter set doesn't exist validation
 68:  Scenario: This object step definition text parameter exists validation
 89:  Scenario: This object step definition text parameter exists with other parameters validation
111:  Scenario: This object exists validation
126:  Scenario: This object step definition exists validation
141:  Scenario: This object step definition parameter set exists validation
164:  Scenario: This object step definition text parameter doesn't exist validation
```

The negation pair sits 96 lines apart in the same file. Both worker sessions **read this file** (call 13 in each timeline). The data is in context.

The miss is therefore not data availability — it is the prompt failing to direct Claude to **scan against** the negation pairs before editing. `fix.md` directs Claude to use `${shortlistPath}` and the UML interaction specs to find the *classes* to edit; it says nothing about scanning the *scenarios* in the runner's own `.feature` to spot pairs whose pass-state the edit must preserve. R52 read the file, focused on the failing TC, edited a code path that satisfies it, and submitted — without re-scanning the sibling list. R53 wrote a guarded implementation that happened to preserve the sibling, but largely by aligning on the existing `validateStepParameterListWorkspace` pattern, not by an explicit sibling-check pass.

The Phase Verification check then runs the **whole** runner — every `@RGR` Test-Case — so R52's edit fails on the sibling and the build retries.

Place: **prompt** (`fix.md` behavioral instruction; no new data file is needed because the `@RGR` tag already scopes the sibling set).

# The Fix

Proposed scope (no issue filed yet — propose filing one):

1. **`fix.md` behavioral instruction.** Add one paragraph telling Claude that the runner is tag-driven and every sibling Test-Case sharing the runner's tag is part of the Verify check. Recommend the worker scan the runner's `.feature` for negation pairs (`X exists` / `X doesn't exist`, `is Y` / `isn't Y`, `with` / `without`) before submitting an edit. The runner's tag is already known to `GreenPhase` (used to build the runner-class name) and can be templated in as `${runnerTag}`.
2. **No new precompute file.** The `@RGR` tag scopes the sibling set; the `.feature` file already lists the siblings in one block and is already in Claude's read history. Adding a precomputed `sibling-scenarios.md` would re-materialize data that is intrinsic to the tag-filtered runner.
3. **No code-side enforcement.** The Verify check already catches the regression; the goal is to shave the *retry cost* (1m 34s on this run) by getting it right on the first attempt.

The fix is in the prompt's *behavioral framing*, not in any precompute or data shape. R52 had the sibling visible in its own context window and still missed it — that is the gap to close. Tightening the directive turns a passive read into an active check.

This is **orthogonal** to [#404][4]. [#404][4] addresses search-strategy variance via the JaCoCo coverage shortlist; this addresses sibling-regression variance via prompt phrasing. The two fixes stack independently — #404 narrows the where-to-edit search, this narrows the will-the-edit-regress-a-sibling check.

# Open Questions From This Case

- The 1m 24s search-strategy variance is independent of the 1m 34s retry variance. Does a `fix.md` negation-pair instruction help on the retry only, or does framing the runner as tag-driven also shorten exploration (Claude may grep less if it knows the `.feature` covers all siblings)? Re-running the same scenario after the prompt edit would tell.
- R52's commit ships a `TestStepIssueTypes` enum value with description text that **duplicates** an existing enum's description. The refactor phase did not collapse the duplicate. A separate study could check how often green-phase additive edits leave duplicates that refactor doesn't fold — that may be a third remediation site (refactor-prompt awareness of enum-value collisions).
- The two-prompt split (`identify.md` then `fix.md`) is unchanged from [pbc-4849][5]. On this case, identify ran short (40–66 s) and added no signal; fix carried 100% of the variance. If a future case shows identify-side variance again, collapsing back into one prompt may be re-litigated.
