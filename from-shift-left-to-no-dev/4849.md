# Context

This page is a worked comparison in the same tradition as [pbc-4445][3]. One Darmok scenario ran on consecutive days against `sheep-dog-grammar`, on branches `Rebuild48` (May 23) and `Rebuild49` (May 24). The pair-range was wide enough to inspect; the diagnosis traces to the same prompt-template site as the [pbc-4445][3] cases but with a sharper proximate cause — the prompt asks Claude to mine a JaCoCo annotation (`title="<test-name>"`) that the project's pom does not produce.

| | Case |
|---|---|
| Scenario | `This object step definition parameter set doesn't exist validation` |
| Rebuild48 commit | `965a81ab` |
| Rebuild49 commit | `dfa011f1` |
| Green-phase range | **2.0 min** (121,761 ms) |
| Files patched | (same — new feature path, no main-code patch in green) |
| Diagnosed cause | prompt — JaCoCo cross-reference asks for data the report does not contain |

The remediation is filed as [issue #404][4]: precompute the JaCoCo coverage shortlist inside `GreenPhase` and have `identify.md` reference the resulting file rather than scrape per-line `title=` attributes that don't exist.

The divergence is at **command #6**. R49 spent ~50 s on commands #6–#15 chasing the missing annotation. R48 reached the same fallback in ~12 s.

# Root Cause

The `title="<test-name>"` annotation the prompt instructs Claude to extract **does not exist in this project's JaCoCo reports**. Verified by re-running `mvn clean verify` on `sheep-dog-grammar` and inspecting a fresh `*.java.html`:

```
attribute names present: charset, class, content, encoding, equiv, href,
                         id, lang, onload, rel, src, type, version, xmlns
```

Only `class="fc"` / `class="nc"` (full/no-coverage span markers) and `id="Lxx"` (line numbers). No per-line test attribution.

`jacoco-sessions.html` confirms why: exactly one session is recorded —

```
Session: ubuntu-client-f6c99eb5
Start:   May 24, 2026, 6:22:19 PM
Dump:    May 24, 2026, 6:22:25 PM
```

The pom configures the standard `prepare-agent` execution + an Ant `ReportTask` over a single `jacoco.exec`. Per-test attribution requires multiple sessions (one execution-data file per test, aggregated). The directory is named `jacoco-with-tests` (aspirational) but the config produces aggregate-only coverage.

The aggregate coverage data **is** still useful for the prompt's stated goal — narrowing Claude's search to the classes the failing test's family actually touches, rather than reading the whole tree. The available signal is: production classes with non-zero coverage, sorted by which packages the runner exercises. That signal lives in `jacoco.xml` (line-level `<line nr="N" mi=".." ci=".."/>` entries inside `<sourcefile>` blocks) and `jacoco.csv` (class-level totals).

# The Fix

[Issue #404][4] — precompute a coverage shortlist in `GreenPhase` and reference it from `identify.md`.

Concretely:

1. `GreenPhase` parses `target/site/jacoco-with-tests/jacoco.xml` once before invoking Claude.
2. Writes `target/darmok/jacoco-shortlist.md` containing:
   - Production classes in the runner-class's package family, sorted by `covered/missed` instruction ratio, with per-class covered line ranges
   - Sibling test classes that heavily cover those production classes
3. `identify.md` step 2 changes from "grep `title=` attributes in `*.java.html`" to "read `${darmokDir}/jacoco-shortlist.md` and use the listed classes as the search shortlist."
4. The UML/source-tree fallback paths stay in the prompt for the compile-error case where `jacoco.xml` may be absent.

The change is in the system, not in the worker. The expectation is that the +60–90 s of identify-phase variance attributable to the missing-annotation chase disappears across runs.

Collapsing `identify.md` + `fix.md` into one green prompt is **not** in scope of #404. That decision is deferred until the shortlist's wall-clock impact has been measured — the two-phase split also serves as a "diagnose-before-edit" guardrail independent of search cost.

# Mapping to the Research

| Predicted ([pbc-research][2]) | Observed |
|---|---|
| Wide pair-range | green-phase +2.0 min on a 4–9 min baseline |
| Each run individually within in-control band | yes — both within typical green-phase range for this scenario |
| Both runs pass the test | yes |
| Two work-trees differ | not in this case — green produced the same artifacts; the variance is entirely in *how long it took to get there* |

The signature differs from [pbc-4445][3] in one important way: the [pbc-4445][3] cases had observable artifact differences (different files patched in case 1, different code styles in case 2). The Rebuild48–49 case has **no work-tree difference at all** — same files, same code. The variance is purely in the *path* the worker took through the prompt. The signal still fired at 2.0 min on the same prompt step.

This strengthens the [pbc-4445][3] claim rather than weakening it. A range chart that only flagged artifact-differing pairs would miss pure-path variance like this one; the pair-range signal catches it the same way.

## green time pair range

This was a two-run comparison (Rebuild48 vs Rebuild49) with a wide green-phase pair-range of 2.0 min (121,761 ms) on a 4–9 min baseline. The diagnosis was a prompt/producer cause: `identify.md` instructs Claude to mine a `title="<test-name>"` JaCoCo annotation that the project's pom never produces, so each run made progressively narrower bash queries chasing a missing attribute and diverged at command #6 — one run bailed to a source-tree fallback in ~12 s, the other spent ~50 s widening across more `.java.html` files. Pair-range caught this even though both runs produced identical artifacts (same files, same code); the variance was purely in the path the worker took through the under-specified prompt step. The fix is in the producer ([#404][4]): precompute a coverage shortlist from `jacoco.xml` and have `identify.md` read it rather than scrape attributes that don't exist.

# Open Questions From This Case

- The [pbc-4445][3] open question about exploration-depth signatures (`Agent` subagent count, distinct files read, `xargs grep` cross-reference count) holds here too. Rebuild48 and Rebuild49 used no subagents; the divergence is in *bash variation depth*. A "bash retries on the same target file" counter might be the cheap signal for this prompt-shape case.
- After [#404][4] lands, re-running this scenario 2–3 times on the same branch will show whether identify-phase variance narrows. If it does not, the next assignable cause becomes visible (likely: fix-phase Claude-turn-count variance, currently the +16 s minor signal).
- The remaining refactor-phase variance (+173 s in this pair) is out of scope here but worth a parallel study. The post-compact session for R49 ran a full Phase 2 behavioral analysis on a new file (`RowIssueDetector.java`); R48 short-circuited at Phase 1 on a structural rename. Different cause site, different prompt — separate investigation.
