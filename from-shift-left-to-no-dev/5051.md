# Context

This page is a worked comparison in the tradition of [pbc-4445][3] and [pbc-4849][4]. The same Darmok scenario — `This object step definition parameter set doesn't exist validation` — ran on `sheep-dog-grammar` on consecutive nights, `Rebuild50` (May 24 → 01:21 EDT on May 25) and `Rebuild51` (May 25 → 11:46 EDT on May 25). The pair-range was the widest on the gsheet (gid 1236418776) for this pair across all 12 active scenarios. The diagnosis returns to [pbc-4445][3] case 1's signature — different files patched, both runs pass — on the same scenario that [pbc-4849][4] tracked to a prompt-template cause and that [pbc-4445][3] case 2 saw converge on the same files.

| | Case |
|---|---|
| Scenario | `This object step definition parameter set doesn't exist validation` |
| Rebuild50 commit | `6decf5dd` |
| Rebuild51 commit | `790751de` |
| Green-phase range | **2.79 min** (166,869 ms) |
| Files patched | **different** between branches (TestStepIssueDetector vs RowIssueDetector) |
| Diagnosed cause | prompt-reference gap — `identify.md` cites `${umlDir}/uml-interaction.md` which doesn't exist; the explicit `RowIssueDetector.validateCellListWorkspace` example lives in `uml-interaction-main.md`, which neither run opened |

This was the widest green-phase range on the gsheet for the Rebuild50–51 pair (range 166,869 ms; next-widest was Header row Cell names at 100,207 ms). The same scenario topped the previous pair's range in [pbc-4849][4] at 121,761 ms. Three consecutive case studies on this scenario — pbc-4445 case 2 (same files), pbc-4849 (same files), pbc-5051 (different files) — show the scenario sits on a spec-ambiguity boundary that fires intermittently.

# Root Cause

The diagnosis is **prompt-driven anchoring on the wrong abstraction**, *not* spec ambiguity. A Cucumber test cannot specify which implementation site a fix lives at — that's by design, the test asserts on the observable diagnostic only. The constraint that *does* specify it lives in the UML interaction spec, and it's not being read.

## The convention is documented

`site/uml/uml-interaction-main.md` lines 92–149 explicitly specify both halves of the answer R51 reached by inference:

- **Lines 92–110 (`## {Type}IssueDetector` section)** — the framework convention is **one detector per cursor type**. The `ValidateActionImpl` cascade is shown as an `instanceof ITestStep` branch dispatching to `TestStepIssueDetector.validate{StepObject,StepDefinition}Name{Only,Workspace}`. By the section's framing, an `IRow` cursor demands a `RowIssueDetector` — the `{Type}` placeholder maps to whatever model class the cursor is.
- **Lines 143–149** spell out `RowIssueDetector.validateCellListWorkspace` *by name*, with a verbatim code example showing the exact ancestor-navigation chain (`getAncestor(theRow, ITable.class) → getAncestor(table, ITestStep.class)`) and the `getCellListAsString` comparison. This is what R51 reinvented from scratch.
- **Lines 129–138** also show `RowIssueResolver.correctCellListWorkspace` (the quick-fix half), confirming the IRow-owns-this-concern pattern across detector + resolver.

So the answer to "which detector should validate a row-cell mismatch?" is unambiguously specified: **`RowIssueDetector`**, by class name, with a working code template.

## The prompt doesn't point at the right file

[identify.md][5]'s Inputs section lists `**UML interaction spec**: ${umlDir}/uml-interaction.md` — but **that file does not exist on disk**. The actual files are `uml-interaction-main.md` and `uml-interaction-test.md` (split by production-code patterns vs test patterns). When claude attempts the Read, the path 404s. From there:

- **R50** (call #6) attempted `uml-interaction.md`, got nothing, eventually globbed and read `uml-interaction-test.md` (call #19) — but never read `uml-interaction-main.md` where the answer lives. With no spec guidance, R50 fell back to pattern-matching the *method list* it could see inside `TestStepIssueDetector.java` and concluded the missing 5th method belonged on that class.
- **R51** (call #8) attempted the same 404, also read only `uml-interaction-test.md` (call #10), never read `uml-interaction-main.md`. R51's correct conclusion came from a *separate* convention it inferred by reading `ValidateActionImpl.java` (call #14) before opening any detector class — the cursor `instanceof` branches read like a one-detector-per-type dispatch table, and the absence of an `instanceof IRow` branch combined with the failing scenario's `rowList/1` cursor path made the answer fall out.

R51 was right *by luck of read-ordering*, not because the spec told it. Run it again with a different read sequence and R51 could just as easily land on R50's path.

## Earlier same-scenario pairs converged because the codebase didn't tempt them

This is the same scenario [pbc-4445][3] case 2 and [pbc-4849][4] both tracked. Those earlier pairs converged on the `Row*` path because either `TestStepIssueDetector` didn't yet have surface area that made the `validateStepParameters` extension plausible, or the model in those runs happened to read in R51's order. Either way, the underlying gap — the prompt failing to land claude on `uml-interaction-main.md` — has been latent across every prior case study on this scenario. It surfaced now because the pattern-matching coin landed differently.

# The Fix

The fix sits in the **prompt + UML file layout**, not in the test (which by construction can't specify implementation choice). Three reachable changes, ordered by cost:

1. **Fix identify.md's UML reference path.** Change `${umlDir}/uml-interaction.md` to `${umlDir}/uml-interaction-main.md` (the file the answer actually lives in), or glob the directory (`${umlDir}/uml-interaction*.md`) so claude reads both halves. Smallest change; closes the immediate gap. Would require a corresponding fix to fix.md if it also cites the singular path.

2. **Rename / consolidate** the UML interaction files so `uml-interaction.md` exists. Options: (a) rename `uml-interaction-main.md` → `uml-interaction.md` and `uml-interaction-test.md` stays the same; (b) merge both into a single `uml-interaction.md`. Either way the prompt's existing path resolves. (b) is simpler but loses the main-vs-test split that has its own value.

3. **Add an explicit ordering hint in identify.md.** *"Before opening any `{Type}IssueDetector` class, read the UML interaction spec's `{Type}IssueDetector` section AND read `ValidateActionImpl` (or equivalent action-dispatch entry point) to identify which model node type owns the new check."* Belt-and-braces — works even if the file rename happens later.

Pbc-4849's recommended fix (precompute JaCoCo shortlist, issue 404) addressed a different gap — *which classes to study for prior-art templates*. Pbc-5051's gap is upstream: which *spec file* tells claude where the check belongs in the first place. (1) and (2) are independent of 404 and complement it.

A follow-up question worth asking: **why didn't the refactor phase catch the divergence?** Refactor reads the UML class-level specs (`uml-class-*.md`) and is expected to flag deviations from the documented pattern. If `RowIssueDetector` is documented in `uml-class-*.md` and refactor didn't push R50's `TestStepIssueDetector` deviation back to the spec-conformant shape, that's a separate refactor-prompt gap worth a parallel investigation. R50's refactor phase was 2:52 wall-clock; if it did walk the spec, it apparently didn't escalate the mismatch.

# What This Validates

1. **The pair-range signal continues to fire on this scenario across rebuild generations.** Three consecutive case studies on `parameter set doesn't exist`: pbc-4445 case 2 (4.2 min, same files), pbc-4849 (2.0 min, same files), pbc-5051 (2.79 min, different files). The scenario sits on a stable ambiguity boundary.
2. **Same-file vs different-file output is still not a sufficient disambiguator.** Pbc-5051 differs in patched-file family from both its prior pairs on the same scenario — yet the pair-range signal fired the same way, with comparable magnitude. The chart is robust to which artifact-shape the variance takes.
3. **JSONL bash/grep-timeline reconstruction stays diagnostic in one investigation pass.** The 15 vs 4 Grep ratio + the exact divergence at call #6 (`Read uml-interaction.md` vs `Glob target/site/jacoco-with-tests/`) was extractable from `jq` in a single survey; from there the file-list diff (TestStep* vs Row*) made the spec-ambiguity diagnosis immediate.
4. **The fix site is the prompt, not the test.** A Cucumber test cannot specify implementation choice — that's by design. The constraint that should specify it (`RowIssueDetector` by name, with code template) is in `uml-interaction-main.md`, but identify.md cites a non-existent `uml-interaction.md`. The fix is to make the prompt point at the file the answer actually lives in. This sits in the same fix-site bucket as pbc-4849 (prompt-level) but the gap is upstream — *which spec file gets read*, not *what data the spec provides*.

5. **Read ordering inside identify matters more than the prompt currently acknowledges.** R50 and R51 ran the same prompt against the same codebase on consecutive nights and arrived at different abstraction boundaries because of the order they opened files. R51's correct answer came from reading `ValidateActionImpl` before any IssueDetector class — the cursor-branch convention shaped its mental model first. R50 read `TestStepIssueDetector` first and anchored on its method list. A prompt nudge ordering the reads (entry-point dispatch first, then detector classes) would have steered R50 down R51's path even without fixing the UML reference.

# Open Questions From This Case

- Did `TestStepIssueDetector` gain new surface area between Rebuild45 (the last "same-files" pair on this scenario) and Rebuild50 that made it a more plausible target? A `git log --follow` on the file across that window would answer it; if yes, the original spec ambiguity was hidden by lack of alternatives, and recent code growth surfaced it.
- After the test-tightening fix lands (whatever issue number that becomes), re-running this scenario 2–3 times on the same branch will show whether the pair-range distribution narrows below the assignable-cause threshold. If it does not, the next assignable cause becomes visible (likely model-of-the-day variance on which detector class to invent).
- The +77 s refactor delta (R51 actually *longer* refactor at 4:09 vs R50's 2:52) is in the opposite direction of the green delta — worth a parallel study. R51's refactor likely spent extra time because it had **new** files to inspect against UML specs (RowIssueDetector / RowIssueTypes); R50's edits were in existing classes the refactor walker had cached models for. Different cause site, different prompt — separate investigation.
