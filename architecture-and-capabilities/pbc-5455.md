# Context

This page continues the [pbc-4445][3] / [pbc-4849][5] / [pbc-5253][6] series. The same Darmok scenario ran on consecutive cycles against `sheep-dog-grammar`, both on 2026-05-26, on branches `Rebuild54` and `Rebuild55`. The pair-range topped the gsheet tab (`gid=1837142131`) for the run window.

The investigation led to a **verified harness bug**: the JaCoCo coverage shortlist producer added in [#404][4] (commit `abc5e3f`, 2026-05-25) has never written a single byte. The Java parser fails on every call with `FileNotFoundException: report.dtd`, caught silently and returned as `null`. The narrowing artifact that `fix.md` step 1 directs Claude to read does not exist on disk, on any run, since the feature shipped.

| | Case |
|---|---|
| Scenario | `No component in the first step triggers an error validation` |
| Rebuild54 commit | `41cf7898` |
| Rebuild55 commit | `e7955df9` |
| Green-phase range | **3.2 min** (194,411 ms) — top of `gid=1837142131` |
| Files patched (final, post-refactor) | R54 and R55: **identical** — same two files, same enum name, same detector logic |
| Diagnosed cause | harness — `DocumentBuilderFactory` default loads external DTDs, JaCoCo's `report.dtd` not shipped in report dir, parse throws, silent catch returns `null` |

Both runs arrived at the **same final fix**: an additional enum value `TEST_STEP_CONTAINER_STEP_OBJECT_NAME_ONLY` in `TestStepContainerIssueTypes` and a guard in `TestStepContainerIssueDetector.validateNameOnly` that returns the new description when the first step's component is empty. There is no spec ambiguity here and no sibling-test retry — pure path variance, attributable to the missing shortlist.

A meta-observation from this case: **a buggy harness shows up as a noisy baseline**, not as an outright run failure. Both runs passed, both committed, both produced equivalent code. The methodology's signal is the *pair-range itself* — once it widens past the in-control band, the cause is somewhere in the system the runs share. Here that "somewhere" was an unfixed JAXP default in the harness's coverage-shortlist producer, masking a feature that shipped but never actually executed. The case-study series doesn't just catch model variance; it catches infrastructure variance that the build green-light wouldn't.

## The post-mvn refactor cycle

Both runs reach a green mvn-verify pass on the first Edit batch. Both then proceed: Skill `rgr-refactor forward sheep-dog-grammar` → `validate_main.py` → finds one violation: the enum name does not match the `{TYPE}_{ASSIGNMENT}_{SCOPE}` convention required by the UML `uml-class-TypeIssueTypes.md` regex. Both runs rename to the same compliant name:

| | R54 first-Edit enum | R55 first-Edit enum | Final enum (after refactor cycle) |
|---|---|---|---|
| Name | `TEST_STEP_CONTAINER_FIRST_STEP_COMPONENT_ONLY` | `TEST_STEP_CONTAINER_FIRST_STEP_COMPONENT` | `TEST_STEP_CONTAINER_STEP_OBJECT_NAME_ONLY` |
| Description | `"The first step must have a component"` | `"The first step must have a component"` | `"The first step must have a component"` |

R54 wrote it once with the `_ONLY` suffix (which still failed the assignment-segment regex because `FIRST_STEP_COMPONENT` is not in the valid assignment set); R55 wrote it without `_ONLY` (also failed for the same regex reason). The final commit diffs are equivalent: same two files (`TestStepContainerIssueTypes.java`, `TestStepContainerIssueDetector.java`), same enum name, same detector logic shape. The pair-range is therefore not driven by spec ambiguity, search strategy producing different fixes, or sibling-test regression — it is **pure exploration overhead before the first byte of the same edit**.

# Root Cause

Both runs called `Read /home/farhan/git/sheep-dog-main/sheep-dog-core/sheep-dog-grammar/target/darmok/jacoco-shortlist.md` and both received:

```
File does not exist. Note: your current working directory is /home/farhan/git/sheep-dog-main.
```

The shortlist is the artifact `fix.md` step 1 directs Claude to read first — "if populated it narrows your search to the classes other passing tests in this family already exercise plus the production classes those tests cover." It is produced by `GreenPhase.computeJacocoShortlist(targetDir)`, which parses `target/site/jacoco-with-tests/jacoco.xml` and writes a single-table list of classes with covered line ranges. The producer was added in commit `abc5e3f` on 2026-05-25 — before both Rebuild54 and Rebuild55 (2026-05-26), so the code was present in the build.

The mojo log on 2026-05-26 has **zero occurrences** of `Green: Writing jacoco-shortlist.md`:

```
$ grep -c "Green: Writing jacoco-shortlist" /logs/darmok.mojo.2026-05-26.log
0
```

`GreenPhase.computeJacocoShortlist` returned `null` for every scenario that day. The producer has three `null` exits:

1. `if (!Files.exists(jacocoXml)) return null;`
2. `catch (Exception e) { return null; }` — silently swallowed parse failure
3. `if (classCoveredLines.isEmpty()) return null;`

`jacoco.xml` is present on disk (517 KB on the case-study run's checkout; produced by the red phase's `mvn verify -Dtest=<RunnerClass>`, since the pom binds jacoco-ant's `report-with-tests` execution to the `verify` phase). The third path can't fire either — `sheep-dog-grammar` has 129 sourcefiles and many covered lines after a green test run, not zero.

The actual failing branch is the **silent catch**, reproduced directly against the on-disk file:

```
$ cat TestJacocoParse.java
import javax.xml.parsers.DocumentBuilderFactory;
import java.io.File;
public class TestJacocoParse {
    public static void main(String[] args) throws Exception {
        var doc = DocumentBuilderFactory.newInstance().newDocumentBuilder()
            .parse(new File(args[0]));
        System.out.println("OK");
    }
}

$ java TestJacocoParse .../target/site/jacoco-with-tests/jacoco.xml
PARSE FAILED: java.io.FileNotFoundException:
  .../target/site/jacoco-with-tests/report.dtd (No such file or directory)
```

JaCoCo's XML report opens with a `DOCTYPE`:

```xml
<!DOCTYPE report PUBLIC "-//JACOCO//DTD Report 1.1//EN" "report.dtd">
```

`report.dtd` is a **relative system identifier**. Java's default `DocumentBuilderFactory` resolves it next to the XML file. JaCoCo doesn't ship `report.dtd` in the report directory (`ls target/site/jacoco-with-tests/` confirms — `index.html`, `jacoco.csv`, `jacoco.xml`, per-package directories, and nothing else). The parse throws `FileNotFoundException` on every call. The default JAXP parser would do this on every machine — Linux, Mac, Windows, CI, local. It has been throwing on every Darmok green-phase since [#404][4] landed.

The exception was swallowed by `catch (Exception e) { return null; }` at the bottom of the method, and the surrounding code branches on `if (shortlist != null)` — so `null` silently means "no shortlist this round, no log entry, fall back to UML guidance." That fallback is what `fix.md` is told to lean on when the shortlist is empty; the prompt has no way to distinguish "shortlist absent because nothing's covered" from "shortlist absent because the producer threw."

## Why every scenario, not just one

There is no per-scenario filtering in `computeJacocoShortlist` — it parses whatever `jacoco.xml` is at `target/site/jacoco-with-tests/jacoco.xml` and emits a coverage table for the whole run. The DOCTYPE is identical across jacoco's output, so the parse fails the same way on every call. The producer is a per-run no-op disguised as a per-scenario no-op.

## Why this wasn't caught by tests

The producer's unit-test coverage doesn't exercise the actual JaCoCo XML output — it would have needed an end-to-end repro reading a real `jacoco.xml` from a green-phase mvn verify, or a fixture file containing the DOCTYPE. The producer ships, runs, returns `null`, and nothing fails. The only downstream consumer is Claude, which sees "file doesn't exist" and falls back to UML guidance — also not a failure mode that bubbles up.

The way this surfaced was the **pair-range methodology**: a feature shipped to narrow Claude's search across runs, yet the inter-run variance on shared scenarios stayed in the same band. That's the signal — the fix shipped but the variance it was supposed to reduce didn't move.

Place: **harness — producer**. The fix lives in `GreenPhase.computeJacocoShortlist`.

# The Fix

Patch applied (two edits to `GreenPhase.java`):

```java
// Disable external DTD loading on the parser.
DocumentBuilderFactory factory = DocumentBuilderFactory.newInstance();
factory.setFeature("http://apache.org/xml/features/nonvalidating/load-external-dtd", false);
factory.setAttribute(XMLConstants.ACCESS_EXTERNAL_DTD, "");
Document doc = factory.newDocumentBuilder().parse(jacocoXml.toFile());
```

And remove the silent catch — let any producer failure bubble up with an ERROR log at the call site:

```java
String shortlist;
try {
    shortlist = computeJacocoShortlist(targetDir);
} catch (Exception e) {
    mojoLog.error("Green: jacoco-shortlist.md producer failed: "
        + e.getClass().getName() + ": " + e.getMessage());
    throw e;
}
```

Reproduced the fix against the on-disk `jacoco.xml`: `PARSED OK, root: report — packages: 16, sourcefiles: 129`. `mvn compile` on `darmok-maven-plugin` succeeds.

The `Files.exists(jacocoXml)` path is also tightened: a missing `jacoco.xml` after the red phase's `mvn verify` ran is a fatal upstream invariant violation (the jacoco-ant `report-with-tests` execution failed to bind, the target dir was wiped between phases, etc.), not a state to silently degrade through. Throws `IllegalStateException` with the resolved path. The `classCoveredLines.isEmpty()` `null`-return is kept — that represents a legitimate "the runner exercised nothing" state, which is data, not a bug. The catch-and-swallow path is removed entirely: "I couldn't parse the file you handed me" is never a state worth pressing on through.

Place: **harness — producer**. No spec change, no prompt change, no test-side change required. The producer now does what it was claimed to do in [#404][4]'s commit message.

# What This Validates

1. **Pair-range methodology catches harness bugs, not just model variance.** Both runs passed, both committed clean code, both produced equivalent artifacts. CI was happy; the gsheet's pair-range column was not. A buggy harness shows up as a noisy baseline — the variance that should have shrunk after [#404][4] didn't, and the only way to spot that was to inspect a pair where the variance was widest. This case validates the methodology's claim that infrastructure variance and model variance are both observable through the same lens.
2. **The narrowing artifact has to actually exist.** [#404][4] added the shortlist on the assumption that narrowing Claude's search closes the [pbc-4849][5]-style variance. This case shows the failure mode where the prompt directs Claude at a narrowing artifact that the producer fails to emit: the variance reverts to the [pbc-4849][5] baseline pattern, and the prompt's wording is helpless. The fix lives in the producer, not the prompt.
3. **Silent producer failures are diagnosable from the JSONL, not the mojo log.** The mojo log was missing the `Green: Writing jacoco-shortlist.md` line, but that absence is hard to spot without the consumer-side signal. The JSONL `Read jacoco-shortlist.md` returning `File does not exist` is the first observable place the failure surfaces — Claude was the canary. The patched producer adds an ERROR-level mojo log on exception so future failures surface in both places.
4. **JAXP-default-DTD-loading is a class of bug worth checking elsewhere.** Any XML producer/consumer in the harness that parses content with a `DOCTYPE PUBLIC ... "X"` system identifier inherits this trap. A grep for `DocumentBuilderFactory.newInstance()` across the codebase is a one-shot way to scan for the same shape — anywhere that calls it without setting `ACCESS_EXTERNAL_DTD` to `""` is a latent same-class-of-bug site.
5. **Identical final commits + wide pair-range is the cleanest no-spec-ambiguity case in the series.** [pbc-4445][3] had different files patched, [pbc-5253][6] had different enum-value strategies, [pbc-4849][5] had identical commits but a clear identify-side narrowing miss. This case isolates the producer-failure mechanism in its purest form: the spec was crisp, the runs converged on the same fix, the only delta was *how far Claude wandered before deciding what to write*.

# Open Questions From This Case

- The patched producer will start logging at INFO ("Green: Writing jacoco-shortlist.md") on the next Darmok run. The follow-up question is whether the resulting shortlist actually shrinks the pair-range on this scenario. Re-running Rebuild56/Rebuild57 after the patch lands would tell — same Test-Case, same model, but `fix.md` step 1 is no longer a no-op.
- A grep for `DocumentBuilderFactory.newInstance()` across the broader sheep-dog tree would surface other JAXP-default-DTD-loading sites. The asciidoc/UML pipeline parses XML in several places; whether any of those hit the same trap is worth a 5-minute audit.
- The patch also makes a missing `jacoco.xml` fatal — an upstream invariant violation rather than a silently-skipped state. Worth a separate scan of red-phase / pom configuration to confirm jacoco-ant's `report-with-tests` execution always fires when red's `mvn verify` runs, so the new throw never surfaces in normal operation.
- The two-prompt split (`identify.md` then `fix.md`) is unchanged. On this case, identify ran short (28–31 s) and added no signal; fix carried 100% of the variance. If a future case shows identify-side variance again, collapsing back into one prompt may be re-litigated.
